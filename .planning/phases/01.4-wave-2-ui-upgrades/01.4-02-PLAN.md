## Phase 01.4 Plan 02: API Sharing Support — Two-Step ACL Pattern + Bundle Wiring

### Objective

Add `sharing` parameter to createCredential, updateCredential, and deleteCredential in api.js. Implement the two-step ACL write pattern required by Splunk's REST API when `sharing === 'user'`. Wire bundle.jsx handlers to pass `data.sharing` from CredentialForm into API calls.

**Purpose:** Enable full sharing parity with legacy — users can create app-scoped, global, or user-scoped credentials. Without the two-step ACL pattern, user-scoped credentials fail silently on Splunk (splunkd requires an initial app-scoped write before transitioning to user).

### Context

- Plan 01 CredentialForm now emits `sharing` in handleSubmit onSave payload
- Legacy password-crud.js patterns at L478-481 (create two-step) and L522-523 (update pre-bump) are the source of truth for the ACL dance
- `flattenCredential` already reads `acl.sharing` field — credentials carry their sharing value through the data layer

### Key Legacy References (password-crud.js)

```javascript
// Create two-step ACL (L478-481):
if (sharing === 'user') {
    await splunkdPOST(aclPath, { 'perms.read': read, 'perms.write': write, sharing: 'app', owner });
}
await splunkdPOST(aclPath, { 'perms.read': read, 'perms.write': write, sharing, owner });

// Update pre-bump (L522-523):
await splunkdPOST(buildAclPath(row), { 'perms.read': read, 'perms.write': write, sharing: 'app', owner });
// ... then final ACL with actual sharing at L545

// Delete ACL bump (L572-576):
sharing: row.acl_sharing === 'user' ? 'app' : row.acl_sharing,
```

3. Add two-step ACL pattern — insert before final ACL POST:
   ```javascript

       });

   ```
